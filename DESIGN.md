## Problem Statement

Resume builders today fall into two camps: pretty templates with dumb AI (Reactive Resume, FlowCV), or powerful AI with no UI and no ATS awareness (ChatGPT copy-paste workflows). Paid tools like Rezi do live ATS scoring but charge $10-30/month and lock PDF export behind paywalls. The open source alternatives (Reactive Resume, ResumeLM, OpenResume) are good but none offer the specific combination of beautiful UI + genuinely smart AI + real ATS optimization + the multi-job variant workflow that real job seekers actually need.

Rese bridges this gap: one master resume, tailored automatically for every job you apply to, with live ATS scoring and AI suggestions, completely open source and truly free.

## What Makes This Cool

The "master resume + per-job variants" workflow. Nobody in the open source space does this. Job seekers apply to 10-30 jobs at a time. Today they manually tweak one resume file over and over, losing track of which version went where. Rese maintains one master resume and generates optimized variants per job description, each with its own ATS score. A dashboard shows: "Software Engineer at Stripe, 87%. Backend Dev at Shopify, 72% (missing: Go, distributed systems)." One click to accept AI suggestions per variant.

The live ATS score updating in real-time as you edit is the visual hook. Green/red indicators, keyword match percentages, a number that goes up as you improve your resume. It turns resume writing from a dreaded chore into something that feels like a game.

## Constraints

- Open source (MIT or similar license)
- Must be truly free for core functionality (no paywall on PDF export)
- AI requires an LLM API (OpenAI/Claude/etc). For v1: user brings their own API key. For future monetization: hosted option with API credits
- Must work for all users (technical and non-technical)
- ATS optimization must be honest. No fake "95% guaranteed" claims. Real optimization = clean formatting, keyword matching, proper PDF export
- Accessibility: WCAG 2.1 AA compliance. Keyboard navigable, screen reader friendly. Resume building should be accessible to everyone, including users with disabilities

## Premises

1. The core differentiator is NOT a new feature nobody has, it's the quality of execution: truly beautiful UI + genuinely smart AI + accurate ATS scoring, all open source and actually free. Plus the multi-variant workflow that no open source competitor offers.
2. "95% ATS score" is a marketing claim, not a technical spec. ATS systems vary wildly (Workday, Greenhouse, Lever, iCIMS). The honest version: single-column layout, clean formatting, keyword optimization against the job description, and PDF export that doesn't break parsing.
3. The AI needs to do real work, not just rephrase text. It needs to extract keywords from job descriptions, map them to the user's experience, suggest quantified achievements, and flag gaps. This requires an LLM API, which means either the user brings their own API key or the builder pays for inference.
4. To compete with Reactive Resume and ResumeLM (which already exist and are good), Rese needs a clear reason to exist. The multi-variant workflow is that reason.

## Approaches Considered

### Approach A: Focused Resume Editor (Minimal Viable)
Three-panel app. Left: paste job description. Center: resume editor with structured sections. Right: live ATS score + keyword suggestions. One clean template. BYOK for AI. No backend needed.
- Effort: M (human: ~2-3 weeks / CC: ~2-3 days)
- Risk: Low
- Pros: Ships fast, core experience testable immediately, low infrastructure cost
- Cons: Single template, BYOK limits non-technical users, no resume import

### Approach B: Master Resume + Job Variants (Ideal Architecture) -- CHOSEN
Everything in A, plus: master resume with per-job variants, dashboard with scores per job, resume import (PDF parse), 3-5 polished templates, backend with auth for saving.
- Effort: L (human: ~6-8 weeks / CC: ~2-3 weeks including integration testing and polish)
- Risk: Medium
- Pros: Unique differentiator, solves real workflow, natural monetization path
- Cons: More complex architecture, needs backend + database + auth, PDF parsing is hard

### Approach C: Browser-First, Zero Backend (Creative/Lateral)
Everything client-side. No auth, no backend. localStorage or JSON export. AI via BYOK. Static site.
- Effort: S (human: ~1-2 weeks / CC: ~1-2 days)
- Risk: Low
- Pros: Zero hosting cost, maximum privacy, simplest open source maintenance
- Cons: No cross-device sync, data loss risk, no multi-variant workflow

## Data Model

```
User (Milestone 2)
  - id: string (uuid)
  - email: string
  - name: string
  - apiKey: string (encrypted at rest)
  - createdAt: datetime

MasterResume (Milestone 2, localStorage in M1)
  - id: string (uuid)
  - userId: string (FK to User)
  - title: string
  - createdAt: datetime
  - updatedAt: datetime
  - sections:
      contactInfo: ContactInfo
      summary: string
      experience: Experience[]
      education: Education[]
      skills: Skills[]

ContactInfo
  - fullName: string
  - email: string
  - phone: string
  - location: string
  - linkedIn: string (optional)
  - website: string (optional)

Experience
  - id: string (uuid, for merge patch targeting)
  - company: string
  - title: string
  - startDate: string (YYYY-MM)
  - endDate: string (YYYY-MM, nullable if current)
  - current: boolean
  - bullets: string[]

Education
  - id: string (uuid)
  - institution: string
  - degree: string
  - field: string
  - graduationDate: string (YYYY-MM)
  - gpa: string (optional)

Skills
  - category: string (e.g. "Languages", "Frameworks", "Tools")
  - items: string[]

JobDescription
  - id: string (uuid)
  - rawText: string
  - pastedAt: datetime
  - extractedKeywords: KeywordSet (populated by one-time LLM call)

KeywordSet
  - hardRequirements: Keyword[]    (weight: 0.35 of total score)
  - preferredSkills: Keyword[]     (weight: 0.20)
  - toolsAndTech: Keyword[]        (weight: 0.15)
  - softSkills: Keyword[]          (weight: 0.10)
  (remaining 0.20 weight = formatting compliance, scored separately)

Keyword
  - term: string
  - category: "hard" | "preferred" | "tools" | "soft"
  - matched: boolean
  - matchedIn: string (optional, which section/bullet matched)

Variant (Milestone 2)
  - id: string (uuid)
  - masterResumeId: string (FK)
  - jobDescriptionId: string (FK)
  - createdAt: datetime
  - masterSnapshot: MasterResume (frozen copy at creation time)
  - overrides: JSON Merge Patch (RFC 7396) applied per-section on top of snapshot
      - Can modify or add bullets. To remove a bullet, set it to null in the patch.
      - Can reorder sections via an explicit sectionOrder: string[] array.
  - atsScore: number (0-100)
  - keywordMatches: KeywordSet (with matched flags updated)

  Variant-Master Relationship:
    - Editing master after variant creation does NOT auto-update variants.
    - User can manually "re-sync": shows a per-field diff (master changes vs variant edits).
    - Conflict resolution: if the same bullet was edited in both master and variant,
      user chooses per-conflict (three-way merge UI, similar to git conflict resolution).
```

## ATS Scoring Algorithm

The score is a weighted sum (0-100), computed entirely client-side with zero LLM calls:

| Component | Weight | What it measures |
|-----------|--------|-----------------|
| Hard requirements | 0.35 | % of hard requirement keywords found in resume text |
| Preferred skills | 0.20 | % of preferred/nice-to-have keywords matched |
| Tools & tech | 0.15 | % of specific tools, languages, frameworks matched |
| Soft skills | 0.10 | % of soft skill keywords matched |
| Formatting compliance | 0.20 | Single column, no tables/images, standard section headings (Contact, Experience, Education, Skills), parseable date formats, text-selectable |

### Keyword Matching

**Fuzzy matching:** "JavaScript" matches "JS", "React.js" matches "React", etc.

Common abbreviation mappings are maintained in a local JSON dictionary at `src/data/keyword-aliases.json`, mapping canonical terms to aliases:
```json
{
  "JavaScript": ["JS", "Javascript", "javascript", "ECMAScript"],
  "TypeScript": ["TS", "Typescript", "typescript"],
  "React": ["React.js", "ReactJS", "react"],
  "PostgreSQL": ["Postgres", "postgres", "PSQL"],
  "Amazon Web Services": ["AWS", "aws"]
}
```

This dictionary ships with the repo and is user-extensible via PRs. The LLM extraction step can also suggest new aliases at runtime, stored in localStorage for the session.

### Two-Stage Architecture

1. **Stage 1 (one-time, on JD paste):** LLM API call extracts and categorizes keywords from the job description. Returns a KeywordSet. Takes 2-5 seconds. Results cached in memory/localStorage.

2. **Stage 2 (continuous, on every edit):** Local deterministic scoring. Compares resume text against extracted keywords using fuzzy matching. Recomputes full score in ~5ms. Triggered 300ms after user stops typing (debounce).

**No LLM in the scoring loop.** This is what makes the "live score" possible. The expensive AI call happens once. After that, scoring is instant.

## Error Handling & Graceful Degradation

| Scenario | Behavior |
|----------|----------|
| LLM API failure (invalid key, rate limit, network) | Error toast. ATS scoring still works if keywords already extracted. Resume editing never blocked. |
| LLM API slow (>10s) | Loading spinner on keyword extraction. All other functionality available. |
| No API key configured | App works as manual resume editor with template + PDF export. ATS scoring and AI suggestions disabled with prompt to add key. |
| Browser storage full (M1) | Warning notification. Offer JSON export as backup. |
| Autosave | Every 10 seconds to localStorage (M1) or database (M2). No user action required. |
| Variant re-sync conflict | Three-way merge UI. User chooses per-conflict. |

## Open Questions

1. **Template engine:** Build custom or adopt an existing open source template system? Reactive Resume's approach vs building from scratch.
2. **PDF export quality:** html-to-pdf (puppeteer/playwright) vs react-pdf rendering? Each has tradeoffs for ATS compatibility.
3. **Resume parsing accuracy:** PDF parsing is notoriously unreliable. How much effort to invest in import vs starting from scratch?
4. **API key management:** For non-technical users, BYOK is friction. When to introduce a hosted option? What's the free tier?
5. **Hosted version enhanced scoring:** The open source version ships the full scoring algorithm. A future hosted version may add enhanced features (e.g., real ATS system test results) as a paid differentiator.
